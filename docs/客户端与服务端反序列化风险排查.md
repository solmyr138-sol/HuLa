# 客户端→服务端 反序列化风险排查

> 排查时间：2026-05-26
> 排查范围：HuLa 客户端（Tauri/Vue）→ HuLa-Server（Spring Boot）所有 `@RequestBody` 请求 DTO

## 1. `GroupMemberMuteReq.mutedUntil` — 已修复（禁言功能）

- **DTO 字段**: `LocalDateTime mutedUntil`
- **问题**: 客户端原来发 `"9999-12-31T23:59:59"` (ISO 带 `T`)，服务端 `LuohuoJacksonModule` 注册了自定义 `LuohuoLocalDateTimeDeserializer`，但 `BaseConfig` 中 `objectMapper.registerModule(new LuohuoJacksonModule()).findAndRegisterModules()` 的调用顺序导致 `JavaTimeModule` 覆盖了自定义反序列化器，`mutedUntil` 静默返回 `null`，数据库 `muted_until` 存为 NULL，禁言拦截永远不生效。
- **修复**:
  - 客户端（`useChatMain.ts`、`GroupPolicySettings.vue`）：日期格式从 `T` 分隔改为空格分隔 `"9999-12-31 23:59:59"`
  - 服务端（`GroupMemberMuteReq.java`）：加 `@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")` 注解
  - 数据库：直接 UPDATE 将现有记录 `muted_until` 设为 `9999-12-31 23:59:59`

## 2. `AdminStatsController` — 管理后台接口（`@RequestParam LocalDateTime`）

- **位置**: `AdminStatsController.getLoginRank()` 和 `getActiveUsers()`
- **字段**: `@RequestParam LocalDateTime start, LocalDateTime end`
- **风险**: `@RequestParam` 走的是 `String2LocalDateTimeConverter`，只支持 `yyyy-MM-dd HH:mm:ss`、`yyyy/MM/dd HH:mm:ss`、`yyyy年MM月dd日HH时mm分ss秒`。**不支持 ISO 带 `T` 的格式**。
- **影响范围**: 仅 HuLa-Admin 后台管理面板调用，**不影响移动端/桌面端 APP**

## 3. 已确认无问题的项目

| 排查项 | 结论 |
|---|---|
| **IM 全部请求 DTO (`*Req*.java`)** | 除 `GroupMemberMuteReq` 外，无 `LocalDateTime`/`LocalDate` 字段 |
| **WS / Base 模块请求 DTO** | 无 `LocalDateTime`/`LocalDate` 字段 |
| **`ChatMessageReq.roomId`** | Rust 端是 `String`，Java 端是 `Long`，Jackson 支持 `"123" → Long` 隐式转换，无问题 |
| **Long 精度溢出** | 当前最大 ID ≈ 1.65×10¹⁴，远小于 JS `Number.MAX_SAFE_INTEGER` (9×10¹⁵)，无精度丢失 |
| **`Long → String` 序列化**（响应方向） | 服务端 `ToStringSerializer` 把 `Long` 序列化为字符串，客户端 TypeScript 类型定义中 ID 已声明为 `string`，匹配 |
| **`LocalDateTime → timestamp` 序列化**（响应方向） | 服务端 `LocalDateTimeToTimestampSerializer` 输出 epoch millis，客户端 TypeScript 中日期字段声明为 `number`，匹配 |
| **`birthday` 字段** | 服务端 `ModifyNameReq.birthday` 是 `String` 类型（非 LocalDate），无反序列化风险 |
| **`BlackAddReq.deadline`** | 是 `Long` 类型（表示分钟数），非日期字符串，无反序列化风险 |
| **`blockUser` 客户端函数** | 函数已定义但全代码库无调用处（死代码），不影响运行 |

## 总结

真正影响 APP 用户的反序列化问题只有 **第 1 项（禁言 `mutedUntil`）**，已修复。第 2 项仅影响管理后台，且取决于管理后台前端的传参格式。其余所有请求 DTO 均无 `LocalDateTime` 字段，不存在同类风险。
